Project Design Document

Kratos Self-Service UI Node

Architecture, page design, and complete workflow for the custom Ory Kratos self-service UI with tenant-aware admin and authorization extensions.

1. Project Purpose

This project is a server-rendered authentication and account-management application built with Node.js, Express, and Handlebars. It acts as a custom UI for Ory Kratos self-service identity flows such as login, registration, recovery, verification, settings, sessions, consent, and logout.

On top of the standard Kratos flows, the project adds a business layer for multi-tenant onboarding, admin APIs, tenant membership tracking, JWT export, and authorization checks backed by PostgreSQL and Keto-style relation tuples.

2. Design Structure

Presentation Layer

Handlebars layouts and views render the screens. Ory Elements Markup components generate the actual auth cards from Kratos flow nodes.

Application Layer

Express routes initialize flows, call Kratos, enforce middleware, and connect UI actions to the backend services.

Identity Layer

Ory Kratos manages identities, self-service flows, sessions, verification, recovery, and browser cookies.

Authorization Layer

PostgreSQL stores tenant mappings, JWT secures admin APIs, and Keto relation tuples model tenant-role permissions.

3. Page Design Model

The UI is organized into a small set of layouts rather than one page per feature style.

  • Auth layout: used for login, registration, recovery, verification, consent, and logout screens.
  • Welcome layout: used for post-login informational pages and navigation-driven screens.
  • Settings layout: used for account settings and profile update flows.
The visual design is mostly data-driven. Kratos returns a flow object with UI nodes, and the server injects those nodes into Ory Elements Markup components, which then render the final forms and controls.

4. Main Components

  • Express bootstrap: configures middleware, view engine, security headers, cookies, body parsing, and route registration.
  • Middleware: validates sessions, detects language, sets favicon metadata, and protects authenticated routes.
  • Kratos client integration: fetches login, registration, settings, and verification flows and performs identity CRUD through admin endpoints.
  • Tenant services: create tenants, assign users to tenants, and store tenant-user-role mappings in PostgreSQL.
  • Keto sync: mirrors tenant role relationships into authorization tuples for access checks.
  • JWT export: converts a valid Kratos session into an admin access token for protected APIs.

5. Complete Request Workflow

Step 1User requests a screen such as /login or /registration.
Step 2The Express route checks whether a Kratos flow ID already exists in the query string.
Step 3If no flow exists, the app redirects the browser to the Kratos browser flow endpoint.
Step 4Kratos creates the flow and redirects back to the UI with a flow ID.
Step 5The route calls Kratos again to fetch the full flow state and UI nodes.
Step 6The server renders a Handlebars view and inserts an Ory Elements card built from the flow nodes.
Step 7The user submits the form. Kratos validates credentials, traits, recovery data, or verification input.
Step 8Kratos sets or updates the browser session cookie.
Step 9Protected pages and admin flows use session checks or exported JWTs for authorization.

6. Authentication Workflow

Login

  1. User opens the login page.
  2. The route creates or fetches a login flow from Kratos.
  3. The page renders login UI nodes with links to signup and recovery.
  4. After successful login, Kratos issues a session cookie.
  5. Authenticated pages use Kratos toSession() or /sessions/whoami to validate the user.

Registration

  1. User opens the registration page and receives a Kratos registration flow.
  2. The form includes tenant-related traits.
  3. The custom registration POST route converts the tenant name into the tenant UUID from PostgreSQL.
  4. The route forwards the normalized form data to Kratos.
  5. After success, the app reads the new session from Kratos.
  6. The app inserts tenant membership into tenant_users and bootstraps authorization data if needed.

Recovery and Verification

  1. Recovery starts a Kratos recovery flow for password reset or account recovery.
  2. Verification starts a Kratos verification flow for email or identity verification.
  3. Both routes render Ory cards from Kratos flow nodes and redirect appropriately on soft errors.

Settings and Session

  1. Authenticated users visit settings or session pages.
  2. The route requires a valid Kratos session.
  3. The app loads the settings flow or session information.
  4. The user can update identity traits or inspect session metadata.

7. Tenant and Admin Workflow

Beyond standard auth, the project supports a tenant-aware operating model.

  1. An admin identity is created either through a bootstrap route or an admin API.
  2. Tenant rows are created in PostgreSQL.
  3. The identity is stored in Kratos with traits such as email, tenant IDs, and roles.
  4. The tenant_users table becomes the main relational source of truth.
  5. Keto relation tuples are created or synchronized so permissions can be checked consistently.
  6. Admin users can export an application JWT from an active Kratos session.
  7. Protected APIs verify that JWT and confirm the actor has the required tenant role.

8. Authorization Workflow

Browser -> Express UI / API -> Kratos public API for browser flows and whoami -> Kratos admin API for identity management -> PostgreSQL for tenants and tenant_users -> Keto for relation tuples -> JWT for admin API authorization

There are two authorization styles in the codebase:

  • Session-based authorization: the app checks a live Kratos session via cookies or session token.
  • JWT-based authorization: the app exports a short-lived admin token and uses it to protect admin APIs.

9. Important Design Decisions

  • Kratos remains the source of truth for identity and session state.
  • The UI is thin and server-rendered, which keeps auth screens consistent with Kratos flow state.
  • Tenant membership is normalized through database mappings instead of only trusting raw traits.
  • Authorization tuples are synchronized externally so tenant-level permissions can be enforced independently.
  • Admin audit middleware supports Oathkeeper headers first and Kratos session fallback second.

10. Summary

In design terms, this is a layered identity platform UI. Express and Handlebars provide the presentation shell, Kratos provides authentication and self-service flow orchestration, PostgreSQL stores tenant relationships, and Keto plus JWT extend the system into a tenant-based admin authorization platform.

The full workflow starts with a browser request, moves through a Kratos flow lifecycle, establishes a session, and then branches into either user self-service screens or tenant-admin operations secured by additional authorization checks.
